### Domain Testing

#### **Domain Testing Process:**

1. **Identify input variables:** For the product detail feature, two key input variables affecting business logic are:
   - `product_id` (parameter passed in the URL).
   - `quantity` (input field for product quantity to add to cart).
2. **Determine domain rules and Equivalence Partitioning:**
   - For `product_id`:
     - Valid partition: Product ID is a positive integer and exists in the database.
     - Invalid partition: ID is a positive integer but does not exist in the DB, or ID is not a number.
   - For `quantity`:
     - Valid partition: Positive integer greater than or equal to 1.
     - Invalid partition: Equal to 0, negative number, decimal number, left blank, or containing non-numeric characters.
3. **Select representative values:** Select specific values within each partition to design test cases.
4. **Establish test cases:** Use the Single Fault Assumption to combine input values. A total of 9 test cases from TC-PRODUCT-DETAIL-001 to TC-PRODUCT-DETAIL-009 were designed.

### AI Gap Analysis

During the process of using AI tools (GitHub Copilot and Antigravity) to design test cases for the Product Detail feature (FR-06), several shortcomings and logic errors on the AI's part were identified:

- **Equivalence Partitioning logic error:** The AI initially defined the valid partition `EP-Q-1` for the quantity to add to the cart as `> 1`. However, according to the requirements specification, the minimum quantity allowed to add is `1` (i.e., `>= 1`). Defining it as `> 1` missed the crucial boundary ON point of `quantity = 1`.
- **Missing test case implementation:** The AI declared the invalid partition `EP-Q-6` (quantity containing alphabetic characters `"abc"`) in the analysis table, but completely missed designing any test case to execute this partition in the initial test case list.
- **Vague expected results:** For the decimal quantity test case (`quantity = 1.5`), the AI provided a vague expected result: "rejected OR rounded". In contrast, the system specification only accepts positive integers, so the correct expected result must be complete rejection.
- **Duplicate test cases between DT and BVA:** The boundary OFF test case (`quantity = 0`) was generated by the AI as a new BVA test case, which completely duplicated the domain testing case `TC-PRODUCT-DETAIL-005`.
- **Omission of integration and security requirements:** The AI missed navigation test scenarios such as Breadcrumb (FR-23), SEO/UI attribute tests like Image alt text (FR-24), scenarios for adding to cart when not logged in, and security testing against XSS attacks (FR-21/SEC-04) when displaying a product name containing malicious HTML tags.

**Root cause:** The AI designed based on heuristics and common templates of standard product detail pages rather than actually reading, understanding, and strictly adhering to the specific project specifications. The AI also had difficulty maintaining consistency between the variable declaration tables and the actual test case lists.

### AI Gap Analysis

For the Coupon Application feature (FR-09), the AI exhibited significant gaps in logical reasoning and boundary testing:

- **Violation of the Single Fault Assumption:** The test case `TC-COUPON-011` initially designed by the AI combined two separate faults (an expired coupon and an order amount below the minimum threshold). This makes debugging and isolating the root cause ambiguous when the test case fails.
- **Discrepancy in representative values:** In the partition for order amount below the minimum threshold (`EP-TOTAL-3`), the representative value was declared as `299,999 ₫`, but in the actual test case `TC-COUPON-006`, the AI used a value of `200,000 ₫`, breaking traceability.
- **Incorrect selection of valid partition test points:** For the valid order amount partition `EP-TOTAL-1`, the AI selected `300,000 ₫` as the representative value. This is actually the boundary ON point for the `SAVE10` coupon rather than an interior point (`IN` point) deep within the partition. This causes unnecessary overlap with BVA.
- **Redundant cases and missing boundaries:** The AI generated redundant test cases (e.g., `TC-COUPON-013` and `TC-COUPON-017`, which did not provide any new coverage) while completely omitting boundary testing for the `BIGBUY` coupon (500,000 ₫ threshold), empty coupon code input (`""`), and the maximum usage boundary for the `VIP100` coupon (max = 2).

**Root cause:** When facing features with multiple co-dependent constraints (such as active duration, minimum order threshold, and user usage limits), the AI tends to suffer from context overload, leading it to combine multiple faults in a single test case, select representative values that overlap with boundary points, and fail to optimize the total number of test cases.

### Domain Testing

#### **Domain Testing Process:**

1. **Identify input variables for coupon creation/management (Admin):**
   - `code`: Coupon code (string).
   - `type`: Coupon type (`percent` or `fixed`).
   - `discount_value`: Discount value (float/integer).
   - `expired_at`: Expiration date (date string).
   - `min_order_amount`: Minimum order amount (float/integer).
   - `max_uses_per_user`: Maximum uses per user (integer).

### Boundary Value Analysis (BVA)

#### **Boundary Value Analysis (BVA) Process:**

1. **Identify boundary variables:** Numeric input fields with a minimum boundary include:
   - `discount_value`: Must be greater than 0 (> 0). Open minimum boundary.
   - `min_order_amount`: Must be non-negative (>= 0). Closed minimum boundary.
   - `max_uses_per_user`: Must be greater than or equal to 1 (>= 1). Closed minimum boundary.
2. **Identify boundary test points (3-value BVA):**
   - **Boundary B1: discount_value (open boundary > 0, step size = 1):**
     - OFF (boundary) = `0` (Rejected).
     - ON (min + 1) = `1` (Accepted).
     - IN (min + 2) = `2` (Accepted).
   - **Boundary B2: min_order_amount (closed boundary >= 0, step size = 1):**
     - OFF (min - 1) = `-1` (Rejected).
     - ON (min) = `0` (Accepted).
     - IN (min + 1) = `1` (Accepted).
   - **Boundary B3: max_uses_per_user (closed boundary >= 1, step size = 1):**
     - OFF (min - 1) = `0` (Rejected).
     - ON (min) = `1` (Accepted).
     - IN (min + 1) = `2` (Accepted).
3. **Map and supplement test cases:**
   - Link boundary OFF points already tested in domain testing.
   - Add ON and IN test cases (from TC-COUPON-ADMIN-018 to TC-COUPON-ADMIN-023) to fully verify boundary behaviors.

### AI Gap Analysis

The Admin Coupon Management feature (FR-17) carries high complexity regarding roles, authorization, and input validation. The AI left behind several gaps in testing:

- **Severe redundancies between DT and BVA:** The AI designed 3 completely redundant test case pairs where the BVA OFF cases duplicated DT Invalid cases. These are: `TC-COUPON-ADMIN-018` duplicated `TC-COUPON-ADMIN-006` (discount = 0), `TC-COUPON-ADMIN-021` duplicated `TC-COUPON-ADMIN-010` (min_order < 0), and `TC-COUPON-ADMIN-024` duplicated `TC-COUPON-ADMIN-011` (max_uses = 0) in the original run.
- **Endpoint Hallucination:** The AI assumed the coupon listing endpoint was `GET /api/admin/coupons`. However, according to the actual API specification, the correct endpoint is `GET /api/coupons`, which handles both client and admin requests based on authorization middleware.
- **Omissions of real-world scenarios:** The AI missed critical test cases such as creating coupons with an expiration date in the past, sending a POST body completely missing the `code` property (rather than just sending `""`), inputting non-integer values for maximum uses, and detailed validation of all fields in the response body.

**Root cause:** The AI relied on generic RESTful API conventions (assuming admin routes are prefixed with `/api/admin/...`), hallucinating endpoints instead of strictly verifying the provided API specifications. It also failed to deduplicate overlapping test cases between Domain Testing and Boundary Value Analysis.

### Boundary Value Analysis (BVA)

#### **Boundary Value Analysis (BVA) Process:**

1. **Identify boundary variables:** Quantity input fields have a strict minimum boundary of >= 1:
   - `add_quantity`: Strict minimum >= 1.
   - `edit_quantity`: Strict minimum >= 1.
2. **Identify boundary test points (3-value BVA):**
   - With a minimum boundary of $min = 1$ and a step size of $1$:
     - **OFF** (min - 1) = `0` (Rejected or automatically normalized to 1).
     - **ON** (min) = `1` (Accepted).
     - **IN** (min + 1) = `2` (Accepted).
3. **Map and supplement test cases:**
   - The OFF boundary points are already tested in domain testing (`TC-CART-MOBILE-003` and `TC-CART-MOBILE-009`).
   - Add test cases `TC-CART-MOBILE-018` and `TC-CART-MOBILE-019` for `add_quantity` boundary (ON, IN).
   - Add test cases `TC-CART-MOBILE-020` and `TC-CART-MOBILE-021` for `edit_quantity` boundary (ON, IN).

### AI Gap Analysis

The Mobile Cart feature (FR-20) requires specific mobile interface testing (mobile viewport, responsive layout) and quantity synchronization. Gaps in the AI's testing for this feature include:

- **Omitting Mobile-Specific Features:** Initially, the AI designed mobile cart test scenarios identical to the desktop cart (FR-07). It missed testing elements unique to or behaving differently on mobile, such as a sticky checkout button, minimized cart drawer/modal, and swipe-to-delete behaviors.
- **Incorrect Assessment of Badge Count Logic:** The AI failed to identify the system's logic error in calculating the cart badge quantity (the badge counted the number of unique product items instead of the sum of their quantities). The AI's original test case merely asserted "badge updated successfully" without verifying that the badge count must equal the total quantity.
- **Lack of Mobile UI Boundary Interaction Testing:** The AI did not design test cases for directly inputting invalid characters using a mobile virtual keyboard (where users might be restricted to numeric input but can still copy-paste special characters).

**Root cause:** The AI lacks visual perception of mobile layouts (mobile viewports) and touch/swipe interactions. It only tested pure logical data flows, thereby missing critical bugs related to the application's actual rendering and responsive behavior.
